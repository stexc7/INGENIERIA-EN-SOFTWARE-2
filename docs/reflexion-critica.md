# Sección 8 — Reflexión crítica

## Proyecto: Salud Familiar (PWA)

> **Nota de actualización:** esta reflexión se redactó cuando la aplicación funcionaba con datos *mock* en el cliente. Posteriormente se incorporó un **backend real** (Express + Prisma + SQLite, autenticación JWT) en `backend/`, cumpliendo el requisito de persistencia en base de datos. La estrategia de resiliencia descrita en la Sección 2 se mantuvo conceptualmente: `localStorage` pasó de ser la única fuente de datos a ser una **caché de lectura** que se sincroniza contra la API real y sirve como respaldo cuando no hay conexión. El punto 1 de la Sección 6 ("Backend real y autenticación robusta") ya está implementado.

## 1. Introducción

El desarrollo de **Salud Familiar** como Progressive Web App constituyó un ejercicio integral de ingeniería de software centrado en el usuario, en el que convergieron decisiones de arquitectura frontend, principios de accesibilidad universal (WCAG 2.1) y estrategias de resiliencia propias del paradigma *offline-first*. A diferencia de un desarrollo web convencional, una PWA orientada a un segmento etario de adultos mayores con comorbilidades (diabetes, artritis, hipoacusia, baja agudeza visual) exige que las decisiones técnicas no se evalúen únicamente por su corrección funcional, sino por su impacto directo en la usabilidad real de personas con limitaciones sensoriales y motrices. A continuación se documenta un análisis crítico del proceso, organizado en los ejes de resiliencia, accesibilidad, experiencia de usuario, dificultades técnicas encontradas y mejoras prospectivas.

## 2. Resiliencia

La resiliencia de la aplicación se abordó desde tres capas complementarias:

**Capa de red (Service Worker).** Se implementó un *service worker* mediante `vite-plugin-pwa` bajo la estrategia `generateSW` de Workbox, que precachea el *app shell* (HTML, JS, CSS e íconos) en la instalación. Esto garantiza que, una vez visitada la aplicación al menos una vez con conectividad, el *shell* completo quede disponible para renderizado sin red. Se configuró `navigateFallback` apuntando a `index.html`, de modo que cualquier navegación *offline* dentro del *scope* de la PWA sea resuelta por el enrutador de React en el cliente en lugar de fallar con un error de red nativo del navegador.

**Capa de datos (persistencia local).** Dado que la aplicación no depende de una API externa —requisito explícito de esta entrega, que utiliza *mock data*—, toda la información de citas, recetas y notificaciones reside en el *bundle* de JavaScript y se sincroniza de forma optimista con `localStorage` a través de un patrón de *Context Providers* (`AppointmentsContext`, `NotificationsContext`). Esto elimina, por diseño, la superficie de fallo más común en aplicaciones offline-first: la divergencia entre estado remoto y estado local. La consecuencia arquitectónica relevante es que la resiliencia de datos aquí no depende del *service worker*, sino de la persistencia sincrónica del navegador, lo cual simplifica el modelo mental pero también evidencia una limitación: en un entorno de producción real, sería necesario introducir una cola de sincronización (*Background Sync API* o similar) para reconciliar cambios contra un backend cuando la conectividad se restablece.

**Capa de percepción del usuario (feedback de estado de red).** Se implementó el hook `useOnlineStatus`, que suscribe los eventos `online`/`offline` del navegador y alimenta un componente `OfflineBanner` de tipo *molecule*, visible de forma persistente mientras la conexión está caída, además de una pantalla dedicada (`OfflinePage`) para el caso límite de navegación sin *shell* cacheado. Esta redundancia es intencional: la resiliencia técnica sin comunicación explícita al usuario resulta insuficiente en un producto dirigido a personas mayores, para quienes la ambigüedad sobre el estado del sistema genera desconfianza en la herramienta.

## 3. Accesibilidad

El diseño de accesibilidad no se trató como una capa cosmética añadida al final, sino como un requisito transversal derivado directamente de los *personas* documentados en la Fase 1 (usuarios con diabetes, uso de lentes correctivos y aparato auditivo). Las decisiones técnicas concretas fueron:

- **Contraste y legibilidad.** Se definió un sistema de *design tokens* (`tokens.css`) con una tipografía base de 18px (superior al estándar de 16px), pesos tipográficos con suficiente contraste de trazo, y una paleta de color verificada para cumplir ratios de contraste AA (4.5:1) sobre fondo blanco, evitando el uso de gris claro sobre blanco —error común que perjudica a usuarios présbitas.
- **Objetivos táctiles.** Todos los elementos interactivos (botones, radio *cards*, ítems de navegación) respetan un tamaño mínimo de 48×48px, en línea con el criterio de éxito 2.5.5 (*Target Size*) de WCAG 2.1, crítico para usuarios con artritis o control motor fino reducido.
- **Semántica y ARIA.** Los formularios (`LoginForm`, asistente de agendamiento) utilizan asociación explícita `label`/`for`, `aria-invalid`, `aria-describedby` para mensajes de error, y `role="alert"` para que los lectores de pantalla anuncien errores de validación de forma inmediata sin depender de la percepción visual del color. El indicador de pasos (`StepIndicator`) del asistente de tres pasos utiliza `aria-current="step"` para exponer el progreso a tecnología asistiva.
- **Navegación por teclado.** Se implementó un *skip link* (`Saltar al contenido principal`) y un estado `:focus-visible` de alto contraste en toda la hoja de estilos global, evitando la práctica —frecuente y perjudicial— de suprimir el contorno de foco por motivos estéticos.
- **No dependencia exclusiva del color.** Los estados de las citas (confirmada, pendiente, cancelada) se comunican mediante texto explícito dentro del componente `Badge`, y el color se emplea únicamente como refuerzo visual secundario, no como único vehículo de información (criterio 1.4.1).

La validación de este eje se realizó mediante Lighthouse sobre el *build* de producción, obteniendo una puntuación de accesibilidad de 100/100, aunque cabe una salvedad metodológica importante: Lighthouse audita conformidad técnica automatizable (contraste, atributos ARIA presentes, nombres accesibles), pero no sustituye una prueba de usabilidad real con personas con discapacidad, que sigue siendo el estándar de validación más riguroso, tal como se evidenció en el Test de Usabilidad Guerrilla de la Fase 1.

## 4. Experiencia de usuario

La arquitectura de componentes bajo **Atomic Design** (átomos, moléculas, organismos, páginas) resultó particularmente adecuada para mantener consistencia visual y de comportamiento entre pantallas, dado que elementos como `Button`, `Input`, `Badge` y `RadioCard` se reutilizan sin variación estilística entre el flujo de autenticación, el asistente de citas y el listado de recetas. Esta reutilización no es meramente una conveniencia de ingeniería: reduce la carga cognitiva del usuario, quien aprende un único patrón de interacción (por ejemplo, cómo luce y se comporta una tarjeta seleccionable) y lo transfiere entre contextos distintos de la aplicación.

El hallazgo más relevante del Test de Usabilidad Guerrilla de la Fase 1 —dificultad de lectura y baja visibilidad de ciertos colores— se tradujo directamente en decisiones de esta fase: aumento del tamaño base de fuente, simplificación de la paleta a colores de alto contraste, y una navegación inferior (*Bottom Navigation*) con etiquetas de texto visibles en todo momento (no solo iconografía), pensada para usuarios que no necesariamente asocian íconos abstractos con su función.

El asistente de agendamiento de citas en tres pasos (especialidad → fecha/hora/lugar → confirmación) se diseñó deliberadamente como un flujo lineal y no como un formulario extenso de una sola pantalla, disminuyendo la carga de decisiones simultáneas y proporcionando retroalimentación de progreso constante, lo cual mitiga la ansiedad de tarea reportada informalmente por los participantes de mayor edad durante las pruebas de usabilidad.

## 5. Dificultades encontradas

El proceso de implementación no estuvo exento de fricciones técnicas propias del ecosistema de herramientas actual:

- **Interoperabilidad de herramientas de build y pruebas.** Vite no incluye un *test runner* propio compatible con Jest; fue necesario configurar `babel-jest` junto con `@babel/preset-env` y `@babel/preset-react` de forma independiente al *pipeline* de Vite, lo que introduce una duplicidad de configuración de *transpilación* (una para desarrollo/producción vía esbuild/Rollup, otra para pruebas vía Babel) que debe mantenerse sincronizada manualmente.
- **Polyfills del entorno jsdom.** El entorno de pruebas `jsdom` utilizado por Jest no expone globalmente `TextEncoder`/`TextDecoder`, requeridos internamente por `react-router-dom` en versiones recientes, lo que provocó fallos de referencia (`ReferenceError`) en la totalidad de las suites que importaban el enrutador hasta que se añadió el *polyfill* correspondiente en el archivo de configuración de pruebas.
- **Precisión en pruebas de accesibilidad.** Las consultas `getByLabelText` de Testing Library realizan coincidencia exacta sobre el contenido textual completo del elemento `<label>`; al incluir un asterisco visual (`*`) para campos obligatorios dentro del propio `<label>`, las pruebas que buscaban el texto exacto ("Usuario") fallaban pese a que la asociación semántica label/input era correcta. Esto ilustra una tensión real entre accesibilidad (marcar visualmente los campos requeridos) y la ergonomía de las pruebas automatizadas, resuelta mediante coincidencia por expresión regular.
- **Evolución de las herramientas de auditoría.** Lighthouse eliminó en sus versiones recientes (v10 en adelante) la categoría puntuada de "PWA" que en versiones anteriores certificaba instalabilidad, *splash screen* e ícono *maskable* de forma automatizada. Esto obligó a verificar manualmente los criterios de instalabilidad (manifest válido, *service worker* activo, iconografía en los tamaños requeridos) en lugar de depender de una puntuación consolidada, evidenciando que las herramientas de auditoría automatizada no son estáticas y su cobertura de criterios varía entre versiones.

## 6. Mejoras futuras

De cara a una eventual continuación del proyecto más allá del alcance académico de esta entrega, se identifican las siguientes líneas de mejora priorizadas:

1. **Backend real y autenticación robusta.** Sustituir el mecanismo de autenticación *mock* (credenciales embebidas en el cliente) por un backend con hash de contraseñas, tokens de sesión (JWT o cookies `httpOnly`) y una API REST o GraphQL para citas y recetas, eliminando la dependencia de `localStorage` como única fuente de verdad.
2. **Sincronización en segundo plano.** Incorporar la *Background Sync API* de Workbox para encolar acciones realizadas *offline* (por ejemplo, agendar una cita sin conexión) y sincronizarlas automáticamente al recuperar la red, en lugar del modelo actual que asume disponibilidad inmediata de escritura local.
3. **Auditoría de accesibilidad automatizada continua.** Integrar `axe-core` o `jest-axe` en la suite de pruebas unitarias para detectar regresiones de accesibilidad en cada *pull request*, complementando la validación puntual de Lighthouse.
4. **Notificaciones push reales.** Migrar el módulo de notificaciones de un listado estático a la *Push API* y *Notifications API* del navegador, permitiendo recordatorios de citas incluso con la aplicación cerrada.
5. **Internacionalización y tamaño de fuente configurable por el usuario.** Aunque la tipografía base ya es superior al estándar, permitir que el propio usuario incremente el tamaño de fuente desde el perfil (persistido como preferencia) daría un grado adicional de control acorde a los principios de diseño inclusivo.
6. **Pruebas de usabilidad formales con población objetivo ampliada.** Extender el Test de Usabilidad Guerrilla de la Fase 1 a una muestra mayor y con instrumentos de medición estandarizados (por ejemplo, System Usability Scale) para cuantificar mejoras entre iteraciones de diseño.

## 7. Conclusión

El desarrollo de Salud Familiar demuestra que la resiliencia técnica (funcionamiento *offline*, persistencia local) y la accesibilidad (WCAG 2.1, semántica ARIA, contraste) no son atributos independientes, sino dimensiones que se refuerzan mutuamente cuando el diseño parte de necesidades reales de usuarios concretos. La arquitectura Atomic Design facilitó la consistencia de experiencia entre pantallas, mientras que las dificultades técnicas encontradas —principalmente de interoperabilidad entre herramientas de *build* y de pruebas— resultaron formativas al evidenciar que el ecosistema de JavaScript moderno, pese a su madurez, todavía exige integración manual entre subsistemas (Vite, Babel, Jest, Workbox) que no siempre están diseñados para coexistir de forma nativa.
